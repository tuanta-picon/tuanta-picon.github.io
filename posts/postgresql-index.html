Tối ưu PostgreSQL: Index không phải là cây đũa thần

TA Tuan Anh · 20 tháng 1, 2026 · ⏱ 14 phút đọc

"Thêm index vào cho nhanh" — câu nói nghe nhiều đến mức trở thành phản xạ khi query chậm. Nhưng index không phải lúc nào cũng giúp ích, thậm chí đôi khi làm mọi thứ tệ hơn nếu dùng sai. Bài này mình chia sẻ cách tiếp cận tối ưu query PostgreSQL một cách có hệ thống.

Bước đầu tiên: đọc EXPLAIN ANALYZE

Trước khi thêm bất kỳ index nào, hãy hiểu query đang làm gì. EXPLAIN ANALYZE cho bạn thấy execution plan thực tế và thời gian thực thi từng bước: 

EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT u.name, COUNT(o.id) as order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE u.created_at > '2025-01-01'
GROUP BY u.id, u.name
ORDER BY order_count DESC
LIMIT 20;

Output sẽ trông như thế này — học cách đọc nó là kỹ năng quan trọng nhất:

Limit  (cost=1247.32..1247.37 rows=20) (actual time=45.231..45.238 rows=20)
  -> Sort  (cost=1247.32..1252.32 rows=2000) (actual time=45.229..45.232 rows=20)
       Sort Key: (count(o.id)) DESC
       -> Hash Left Join  (cost=89.50..1197.50 rows=2000) (actual time=2.103..43.891 rows=5000)
             Hash Cond: (o.user_id = u.id)
             -> Seq Scan on orders o  (cost=0..892.00 rows=45000)  ← ĐÂY LÀ VẤN ĐỀ
             -> Hash  (cost=64.50..64.50 rows=2000)
                   -> Index Scan on users u  (cost=0.29..64.50 rows=2000)

Seq Scan on orders với 45,000 rows — PostgreSQL đang quét toàn bộ bảng. Đây là lúc cần index.

Các loại index và khi nào dùng

B-tree (mặc định)

B-tree phù hợp cho hầu hết trường hợp: so sánh =, <, >, BETWEEN, LIKE 'abc%' (prefix). Đây là index bạn sẽ dùng 90% thời gian. 

CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_users_created_at ON users(created_at)
  WHERE created_at > '2024-01-01';  -- partial index: chỉ index data cần thiết

GIN / GiST — cho full-text search và JSON

-- Full-text search
CREATE INDEX idx_articles_fts ON articles
  USING GIN (to_tsvector('vietnamese', title || ' ' || body));

-- JSONB containment
CREATE INDEX idx_meta ON products USING GIN (metadata jsonb_path_ops);

-- Sau đó query:
SELECT * FROM products WHERE metadata @> '{"color": "red"}';

Composite Index — thứ tự cột quan trọng

CREATE INDEX ON orders(user_id, created_at) có thể được dùng cho:

  • Query lọc theo user_id ✅
  • Query lọc theo cả user_id và created_at ✅
  • Query chỉ lọc theo created_at ✗ (phải là cột đầu tiên)

Nguyên tắc: cột có selectivity cao (nhiều giá trị khác nhau) và thường dùng để filter đặt trước.

Khi nào index KHÔNG giúp ích

  • Low-cardinality columns: cột status chỉ có 3 giá trị (active/inactive/deleted) — index B-tree gần như vô dụng vì 33% row match. Dùng partial index hoặc bitmapOr thay thế.
  • Hàm trên column: WHERE LOWER(email) = '...' sẽ không dùng index trên email. Phải tạo expression index: CREATE INDEX ON users (LOWER(email)).
  • Implicit type casting: so sánh WHERE user_id = '123' (string) khi user_id là integer sẽ bỏ qua index.
  • Bảng nhỏ: với bảng vài nghìn row, sequential scan thường nhanh hơn index scan vì tránh được overhead.

Chi phí ẩn của index

Mỗi index có chi phí:

  • Storage: index B-tree thường chiếm 20-50% kích thước của dữ liệu gốc.
  • Write overhead: mỗi INSERT/UPDATE/DELETE phải cập nhật tất cả index. Bảng nhiều index → write chậm hơn đáng kể.
  • Vacuum cost: index bloat sau nhiều update/delete. Cần monitor và REINDEX định kỳ.

Quy tắc của mình: chỉ thêm index khi đã chứng minh được query chậm, và đo lại sau khi thêm để xác nhận hiệu quả.

Công cụ hữu ích

-- Tìm index chưa bao giờ được dùng
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0 AND indexname NOT LIKE '%pkey%'
ORDER BY schemaname, tablename;

-- Tìm query chậm nhất (cần pg_stat_statements extension)
SELECT query, calls, mean_exec_time, total_exec_time
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

Kết luận

Index là công cụ mạnh nhưng không phải thuốc chữa bách bệnh. Quy trình đúng: đo → phân tích → thêm index → đo lại → xem xét chi phí. Đừng thêm index "phòng hờ" — chúng sẽ ăn tài nguyên của bạn trong im lặng.

Bạn đã từng gặp trường hợp thêm index mà query còn chậm hơn chưa? Mình đã gặp — và bài học đó tốn của mình một buổi debug.
