LLM AI Prompt Engineering

Prompt Engineering thực chiến — viết prompt để LLM làm đúng ý

TA Tuan Anh · 15 tháng 3, 2026 · ⏱ 11 phút đọc

LLM (Large Language Models) như GPT-4, Claude, Gemini đang thay đổi cách chúng ta làm phần mềm. Nhưng ai cũng biết câu chuyện "garbage in, garbage out" — prompt tệ dẫn đến output tệ dù model xịn đến đâu. Prompt Engineering không chỉ là "hỏi đúng câu" mà là một kỹ năng có thể học và cải thiện có hệ thống.

Tại sao prompt quan trọng đến vậy?

LLM là các mô hình xác suất — chúng không "hiểu" theo nghĩa con người, mà dự đoán token tiếp theo dựa trên context. Prompt chính là context đó. Cùng một model, cùng một câu hỏi nhưng cách đặt vấn đề khác nhau có thể cho kết quả chênh lệch đáng kinh ngạc.

Prompt engineering không phải là hack hay mẹo vặt. Đó là ngôn ngữ giao tiếp với một loại hệ thống hoàn toàn mới.

Zero-shot vs Few-shot prompting

Zero-shot: chỉ mô tả task, không cho ví dụ. Phù hợp với task đơn giản hoặc khi model đã được fine-tune cho domain đó. 

Phân loại sentiment của đoạn text sau:
"Sản phẩm này thực sự tuyệt vời, tôi rất hài lòng!"

→ Positive

Few-shot: cung cấp vài ví dụ input-output trước khi đưa ra câu hỏi thực. Cực kỳ hiệu quả khi task có format đặc biệt hoặc cần consistent output. 

Phân loại sentiment. Ví dụ:
Text: "Dịch vụ tệ, chờ mãi không thấy."  → Negative
Text: "Bình thường, không có gì nổi bật." → Neutral
Text: "Nhân viên thân thiện, rất ấn tượng!" → Positive

Bây giờ phân loại:
Text: "Giao hàng chậm nhưng sản phẩm đúng như mô tả."

→ Neutral

Mình thường dùng 3–5 ví dụ. Ít hơn thường không đủ context, nhiều hơn tốn token mà lợi ích tăng thêm ít.

Chain-of-Thought (CoT) — bắt model "nghĩ thành tiếng"

Với các task phức tạp cần suy luận nhiều bước, thêm cụm "Hãy suy nghĩ từng bước" (hoặc "Let's think step by step") vào cuối prompt có thể cải thiện accuracy đáng kể. 

# Không có CoT
Nếu một cửa hàng giảm giá 20% rồi lại tăng 25%, giá có cao hơn ban đầu không?
→ Không (sai!)

# Có CoT
Nếu một cửa hàng giảm giá 20% rồi lại tăng 25%, giá có cao hơn ban đầu không?
Hãy tính từng bước.

→ Giả sử giá ban đầu là 100.
  Sau giảm 20%: 100 × 0.8 = 80
  Sau tăng 25%: 80 × 1.25 = 100
→ Giá bằng ban đầu, không cao hơn. (đúng!)

Với GPT-4 và Claude, CoT đôi khi được kích hoạt tự động bởi model. Nhưng với task phức tạp, luôn nên explicit.

Role prompting — đặt model vào vai trò cụ thể

Giao cho model một vai trò rõ ràng giúp nó "kéo" đúng kiến thức và phong cách phù hợp từ training data: 

Bạn là một senior backend engineer với 10 năm kinh nghiệm về Go và distributed systems.
Review đoạn code sau và chỉ ra các vấn đề về performance và concurrency safety:

[code here]

So với prompt không có role, output thường technical hơn, có chiều sâu hơn và ít bị "hallucinate" kiến thức cơ bản.

System prompt — foundation của mọi chatbot

Với các API như OpenAI, Anthropic, system prompt là lớp instruction nền tảng chạy trước mọi cuộc hội thoại. Đây là nơi bạn:

  • Định nghĩa persona và tone (formal, casual, technical)
  • Đặt ra giới hạn những gì model được phép làm
  • Cung cấp context cố định (tên công ty, sản phẩm, FAQ)
  • Format output mong muốn (JSON, Markdown, plain text)
// System prompt cho một coding assistant
Bạn là một AI assistant chuyên hỗ trợ lập trình viên Go.

Luôn:
- Trả lời bằng tiếng Việt trừ khi được yêu cầu khác
- Code examples phải compilable và production-ready
- Giải thích "tại sao" chứ không chỉ "như thế nào"
- Mention trade-offs khi relevant

Không bao giờ:
- Tạo code có security vulnerabilities rõ ràng
- Suggest deprecated packages

Kiểm soát output format

Khi tích hợp LLM vào hệ thống, output phải parseable. Luôn yêu cầu format cụ thể: 

# Yêu cầu JSON output
Phân tích đoạn text sau và trả về JSON với các field:
- sentiment: "positive" | "negative" | "neutral"
- confidence: số từ 0 đến 1
- key_phrases: mảng tối đa 3 cụm từ quan trọng nhất
- language: ngôn ngữ của text

Text: "Sản phẩm đến nhanh nhưng hộp bị móp, may là hàng vẫn tốt."

Chỉ trả về JSON, không có text nào khác.

Với Claude và GPT-4, bạn cũng có thể dùng structured output mode (JSON mode) ở API level để đảm bảo parse không bao giờ fail.

Các anti-pattern cần tránh

  • Prompt quá mơ hồ: "Viết về AI" → model không biết scope, level, audience. Hãy cụ thể: "Viết một đoạn giải thích transformer architecture cho developer không có background ML, dài khoảng 200 chữ."
  • Nhồi nhét quá nhiều task: một prompt làm nhiều việc cùng lúc thường bị model drop task. Tách thành nhiều lời gọi nếu cần.
  • Không test với edge case: prompt hoạt động tốt với 5 ví dụ đơn giản có thể fail với input thực tế phức tạp hơn. Luôn test với diverse cases.
  • Assume model nhớ context cũ: mỗi API call là stateless. Nếu cần multi-turn, phải tự manage conversation history.

Kết luận

Prompt Engineering là kỹ năng quan trọng cho bất kỳ ai làm việc với LLM — từ product builder đến backend dev tích hợp AI. Iteration nhanh là chìa khóa: viết prompt → test → phân tích failure cases → cải thiện. Và quan trọng nhất: document lại các prompt đã hoạt động tốt, đừng để mất.
