RAG thực chiến — khi LLM biết đọc tài liệu của bạn

TA Tuan Anh · 5 tháng 3, 2026 · ⏱ 14 phút đọc

LLM có một vấn đề cốt lõi: kiến thức bị đóng băng tại thời điểm training. Chúng không biết về tài liệu nội bộ công ty bạn, codebase của bạn, hay sự kiện xảy ra sau cutoff date. RAG (Retrieval-Augmented Generation) là giải pháp — thay vì train lại model (tốn kém), bạn cho model "tra cứu" tài liệu phù hợp trước khi trả lời.

RAG hoạt động như thế nào?

Pipeline RAG cơ bản gồm 2 giai đoạn:

Giai đoạn Indexing (chạy một lần, hoặc khi tài liệu thay đổi):

  • Load tài liệu (PDF, Markdown, HTML, code...)
  • Chia nhỏ thành chunks (đoạn văn bản vừa đủ)
  • Tạo embedding vector cho từng chunk bằng embedding model
  • Lưu vào vector database (Qdrant, Pinecone, pgvector...)

Giai đoạn Retrieval + Generation (mỗi lần user hỏi):

  • Tạo embedding cho câu hỏi của user
  • Tìm top-K chunks có embedding gần nhất (cosine similarity)
  • Đưa các chunks đó vào prompt như context
  • LLM trả lời dựa trên context được cung cấp

Xây dựng RAG đơn giản với Python

Dưới đây là một pipeline RAG tối giản dùng OpenAI embeddings và Qdrant: 

from openai import OpenAI
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct
import uuid

client    = OpenAI()
qdrant    = QdrantClient(":memory:")   # hoặc URL cho production
COLL      = "docs"
EMBED_DIM = 1536  # text-embedding-3-small

# ── Tạo collection ──────────────────────────────────────────
qdrant.create_collection(COLL, vectors_config=VectorParams(
    size=EMBED_DIM, distance=Distance.COSINE
))

# ── Indexing ────────────────────────────────────────────────
def embed(text: str) -> list[float]:
    return client.embeddings.create(
        input=text, model="text-embedding-3-small"
    ).data[0].embedding

def index_chunks(chunks: list[str]):
    points = [
        PointStruct(
            id=str(uuid.uuid4()),
            vector=embed(chunk),
            payload={"text": chunk}
        )
        for chunk in chunks
    ]
    qdrant.upsert(collection_name=COLL, points=points)

# ── Retrieval + Generation ───────────────────────────────────
def ask(question: str, top_k: int = 4) -> str:
    q_vec    = embed(question)
    results  = qdrant.search(COLL, query_vector=q_vec, limit=top_k)
    context  = "\n\n---\n\n".join(r.payload["text"] for r in results)

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content":
                "Trả lời dựa trên context được cung cấp. "
                "Nếu không tìm thấy thông tin, hãy nói rõ."},
            {"role": "user", "content":
                f"Context:\n{context}\n\nCâu hỏi: {question}"}
        ]
    )
    return response.choices[0].message.content

Chunking strategy — điều quan trọng nhất bạn thường bỏ qua

Chất lượng RAG phụ thuộc rất nhiều vào cách bạn chia tài liệu. Không có cách nào đúng duy nhất, nhưng đây là các lựa chọn phổ biến:

Fixed-size chunking

Chia theo số token cố định (ví dụ: 512 token), có overlap giữa các chunk. Đơn giản, hoạt động ổn với văn bản liên tục. 

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=64,          # overlap giúp không mất context biên
    separators=["\n\n", "\n", " ", ""]
)

Semantic chunking

Chia theo ngữ nghĩa — tìm điểm mà embedding thay đổi đột ngột giữa các câu. Phức tạp hơn nhưng giữ nguyên đơn vị ngữ nghĩa, tốt hơn cho tài liệu có cấu trúc.

Document-aware chunking

Chia theo cấu trúc tài liệu: markdown headers, PDF sections, code functions. Mình khuyến nghị approach này cho hầu hết use case thực tế vì nó giữ nguyên context tự nhiên. 

# Chia Markdown theo heading
import re

def chunk_markdown(text: str) -> list[dict]:
    sections = re.split(r'^#{1,3} .+', text, flags=re.MULTILINE)
    headers  = re.findall(r'^(#{1,3} .+)', text, flags=re.MULTILINE)

    chunks = []
    for header, content in zip(headers, sections[1:]):
        chunks.append({
            "header": header.strip("# "),
            "text":   (header + "\n\n" + content).strip()
        })
    return chunks

Đánh giá chất lượng RAG

Làm sao biết RAG pipeline của bạn tốt? Hai metric quan trọng nhất:

  • Context Recall: các chunk được retrieve có chứa thông tin cần thiết để trả lời không? Đo bằng cách so sánh với ground-truth context.
  • Answer Faithfulness: câu trả lời của model có dựa vào context được cung cấp không, hay model đang "hallucinate" thêm thông tin bên ngoài?

Framework RAGAS tự động đánh giá cả hai metric này bằng cách dùng một LLM khác làm "judge": 

from ragas import evaluate
from ragas.metrics import faithfulness, context_recall

results = evaluate(
    dataset=your_test_dataset,
    metrics=[faithfulness, context_recall]
)
print(results)
# {'faithfulness': 0.87, 'context_recall': 0.92}

Các vấn đề thường gặp và cách xử lý

  • Retrieve sai chunk: thường do chunk quá lớn hoặc câu hỏi không match với cách tài liệu được viết. Thử query rewriting — dùng LLM paraphrase câu hỏi trước khi search.
  • Context window bị tràn: top-K chunks quá nhiều token. Dùng reranking (cross-encoder) để chọn chunk liên quan nhất từ top-20 rồi chỉ đưa top-4 vào prompt.
  • Model hallucinate ngoài context: thêm instruction rõ: "Chỉ trả lời dựa trên context. Nếu không có thông tin, trả lời 'Tôi không tìm thấy thông tin này.'"
  • Tài liệu lỗi thời: implement metadata filtering + timestamp. Chỉ retrieve chunk từ tài liệu trong khoảng thời gian nhất định.

Kết luận

RAG là một trong những pattern hữu ích nhất khi xây dựng ứng dụng AI production. Nó giải quyết vấn đề knowledge cutoff và bảo mật dữ liệu (không cần upload tài liệu nhạy cảm lên training pipeline). Bắt đầu đơn giản, đo lường kỹ, rồi mới tối ưu từng layer.

Bạn đang xây dựng RAG cho use case nào? Mình rất muốn nghe kinh nghiệm thực tế của bạn.
