Làm thế nào để viết Pull Request mà reviewer "thích mê"

TA Tuan Anh · 8 tháng 1, 2026 · ⏱ 6 phút đọc

Code review là một trong những thực hành tốt nhất của software engineering. Nhưng review chỉ có giá trị khi Pull Request được viết tốt. Một PR mơ hồ, khổng lồ, không có context sẽ nhận được review qua loa hoặc bị trì hoãn vô thời hạn. Mình đã từng ở cả hai phía — person tạo PR và reviewer — và đây là những gì mình học được.

Kích thước PR: nhỏ thôi, nhưng đủ ngữ nghĩa

Nguyên tắc vàng: một PR — một mục tiêu. Không phải một PR cho cả sprint.

Nghiên cứu từ SmartBear cho thấy: reviewer hầu như không thể review hiệu quả hơn 400 dòng code trong một lần. Quá đó, defect detection rate giảm đáng kể.

  • Refactor riêng → feature riêng → bug fix riêng
  • Nếu thay đổi DB schema, tách thành PR migration trước, PR code sau
  • Dùng feature flag để merge code chưa sẵn sàng mà không ảnh hưởng production

Mô tả PR: đừng để reviewer đoán mò

Đây là template mình dùng cho mọi PR:

## Vấn đề
Mô tả ngắn gọn vấn đề đang giải quyết. Link issue/ticket nếu có.

## Giải pháp
Mình đã làm gì và tại sao chọn cách này thay vì cách khác.

## Thay đổi chính
- [ ] Thêm endpoint POST /api/v2/orders
- [ ] Cập nhật schema: thêm column discount_amount
- [ ] Viết migration và rollback script

## Cách test
1. Chạy `docker compose up`
2. Gọi API: `curl -X POST localhost:8080/api/v2/orders -d '{"item_id": 1}'`
3. Kiểm tra response có trường discount_amount

## Screenshot (nếu có UI)
[Trước] / [Sau]

## Lưu ý cho reviewer
Phần logic tính discount ở file `pricing/calculator.go:45` có thể
gây tranh luận — mình chọn approach này vì lý do X, nhưng muốn nghe ý kiến.

Commit message rõ ràng

Mỗi commit trong PR nên là một đơn vị logic hoàn chỉnh và có thể hiểu độc lập. Mình theo Conventional Commits: 

feat(orders): add discount calculation for premium users

Discount is computed based on the user's tier (silver: 5%, gold: 10%,
platinum: 15%). Applied before tax calculation.

Closes #234

Và những thứ cần tránh:

fix               ← fix gì?
WIP               ← không nên push WIP lên
asdfasdf          ← thực sự mình đã thấy điều này
addressed review  ← cụ thể hơn đi

Tự review trước khi request review

Trước khi assign reviewer, hãy tự review PR của mình như một người ngoài. Checklist của mình:

  • Mình có thể giải thích từng thay đổi không? Nếu không — refactor hoặc thêm comment.
  • Có dead code, debug log, TODO không xử lý không?
  • Tests có cover happy path, edge case và error case không?
  • Có breaking change nào ảnh hưởng consumer hiện tại không?
  • Migrations có rollback script không?
  • Secrets, hardcoded URL, creds có bị commit không?

Review là hợp tác, không phải tòa án

Khi comment, phân biệt rõ: đây là blocking (phải sửa) hay suggestion (có thể xem xét)? Một cách mình hay dùng: 

[blocking] Logic này sẽ lỗi khi user_id là null — cần add nil check

[nit]      Có thể dùng strings.Builder thay vì concatenation cho performance
           tốt hơn, nhưng không bắt buộc trong context này

[question] Tại sao chọn Redis thay vì local cache ở đây? Muốn hiểu thêm

Khi nhận comment, đừng defensive. Nếu đồng ý — sửa và xác nhận. Nếu không — giải thích lý do và thảo luận. Mục tiêu là code tốt hơn, không phải ai thắng.

Kết luận

PR tốt không chỉ là code đúng. Nó là giao tiếp — với reviewer hiện tại và với người maintain code sau này. Đầu tư 15 phút viết mô tả tốt có thể tiết kiệm hàng giờ review và back-and-forth.

Code được đọc nhiều hơn được viết. PR được review nhiều hơn được tạo.

Team bạn có template PR không? Nếu chưa, có lẽ đây là thứ nên đề xuất tuần tới.
